# ApeWorkflow 自我错误上报设计

## 目标

当 ApeWorkflow 在用户实际使用过程中发生可见异常或错误时，自动生成并提交 GitHub issue 到 ApeWorkflow 仓库。

本设计只覆盖自动错误上报，不包含自动修复执行逻辑本身。

## 需求范围

- 覆盖所有用户可见的执行错误和异常。
- 错误一发生就立即上报。
- 同一种错误按错误指纹合并去重。
- 首次出现的指纹创建 issue。
- 再次出现的同一指纹更新已有 issue。
- 目标仓库固定为 ApeWorkflow 仓库。

## 非目标

- 不实现自动修复流程。
- 不实现人工确认后再提交。
- 不将静默日志作为上报对象。
- 不更改现有命令的业务行为和输出格式。

## 现状

仓库里已经存在 GitHub issue 提交能力：

- `src/commands/feedback.ts` 里已经实现了通过 `gh issue create` 提交 issue 的逻辑。
- `src/cli/index.ts` 里各命令入口已经存在统一的 `try/catch` 包装。
- `src/commands/workspace.ts`、`src/commands/context-store.ts` 等模块已经有各自的 `handleFailure`。

当前缺少的是一个统一的错误标准化和上报层。

## 总体方案

新增一个统一的错误上报模块，流程如下：

1. 捕获错误。
2. 将错误转换成标准事件。
3. 计算错误指纹。
4. 查询本地去重状态。
5. 如果指纹未出现过，则创建 GitHub issue。
6. 如果指纹已出现过，则更新对应 GitHub issue。
7. 记录本地状态，供后续去重使用。

## 架构

### 1. 错误捕获层

错误来源分为三类：

- 命令级异常：各命令 `catch` 里捕获到的错误。
- 进程级异常：`uncaughtException`。
- 进程级拒绝：`unhandledRejection`。

这些入口都必须统一转发到上报模块。

### 2. 错误标准化层

错误上报模块将任意 `unknown` 错误标准化成固定结构，至少包含：

- `message`
- `name`
- `stack`
- `commandPath`
- `timestamp`
- `platform`
- `version`
- `workspaceRoot` 或当前执行上下文

标准化后的结构只用于上报和去重，不改变原始错误对象。

### 3. 错误指纹层

错误指纹由稳定字段生成，建议包含：

- 错误名称
- 错误消息
- 堆栈首个稳定位置
- 命令路径
- 相关 workspace 或 context 标识

指纹必须满足：

- 同一类错误稳定一致。
- 不同错误尽量不碰撞。
- 不依赖时间戳。

### 4. 本地去重层

本地保存一个错误索引，记录：

- 指纹
- 对应 issue number
- 对应 issue url
- 首次上报时间
- 最近上报时间
- 上报次数

作用：

- 避免重复创建 issue。
- 允许同一错误后续更新原 issue。

### 5. GitHub 上报层

上报层复用现有 GitHub 提交能力，固定提交到 ApeWorkflow 仓库。

首次出现：

- 创建 issue。
- 写入错误摘要、完整事件信息、上下文信息、指纹。

重复出现：

- 追加新的 issue comment。
- 追加最新发生记录。
- 保留原 issue 作为同类错误聚合点。

## 事件内容

每个 issue 至少包含：

- 错误标题
- 错误指纹
- 原始错误消息
- 原始堆栈
- 触发命令
- 运行环境
- 上次发生时间
- 出现次数

标题建议保持稳定，便于重复错误聚合。

## 触发规则

- 错误一旦进入统一错误入口，立即执行上报。
- 不等待自动修复结果。
- 不等待用户确认。
- 如果上报模块自身失败，只记录上报失败，不吞掉原始错误。

## 与现有代码的集成点

### `src/cli/index.ts`

将各顶层命令的 `catch` 统一转发到错误上报模块。

### `src/commands/workspace.ts`

将 `handleFailure` 中的错误转发到错误上报模块。

### `src/commands/context-store.ts`

将 `handleFailure` 中的错误转发到错误上报模块。

### `src/commands/initiative.ts`

将 `handleFailure` 中的错误转发到错误上报模块。

### `src/commands/feedback.ts`

复用现有 GitHub issue 提交逻辑，作为上报实现的基础能力。

## 配置

需要增加一个开关，用于控制自动错误上报是否启用。

默认值：

- 默认启用。

配置项至少需要支持：

- 是否启用自动上报
- GitHub 目标仓库
- 本地去重状态存储位置

配置存储遵循 ApeWorkflow 现有全局配置目录约定，去重状态文件存放在全局配置目录下。

## 错误处理

### 上报成功

- 记录本地指纹状态。
- 更新 issue 引用信息。

### 上报失败

- 保留原始错误退出行为。
- 不阻断原命令失败结果。
- 不将上报失败误判为业务成功。

### 指纹计算失败

- 回退到基于错误名称、消息和调用点的保守指纹。
- 仍然允许上报。

## 测试

需要补充以下测试：

- 指纹生成测试
- 首次错误创建 issue 测试
- 重复错误更新 issue 测试
- 命令级异常上报测试
- `uncaughtException` 上报测试
- `unhandledRejection` 上报测试
- 上报模块失败不影响原始错误测试

## 验收标准

- 用户可见错误发生后会立即触发上报。
- 同一错误不会重复创建新的 issue。
- 同一错误再次出现时会更新原 issue。
- 现有命令行为不被修改。
- ApeWorkflow 仓库成为唯一默认上报目标。
